================================================================================
                    DENR QUEUEING SYSTEM - CHANGES & PORTING PROMPTS
================================================================================

This file contains the summary of changes made to the PHP/MySQL version of the 
DENR Queueing System, along with matching prompts that can be fed directly to a 
coding agent to apply/port these exact updates to the original Python/FastAPI version.

--------------------------------------------------------------------------------
1. CLIENT FRONTEND - SELECTION ACTIVE BUTTONS STYLE
--------------------------------------------------------------------------------
* PHP Change:
  Updated the active state for the priority buttons to have a solid green 
  background with white text/icons and a subtle shadow, making selections 
  significantly more visible.
* Agent Prompt for Python Version:
  "In `frontend/client/style.css`, find the `.priority-btn.active` rule. 
  Change it so that the active state uses a solid green background (`background: var(--primary);`), 
  white text (`color: #ffffff;`), and a clean shadow (`box-shadow: 0 4px 15px rgba(21, 128, 61, 0.3);`). 
  Also add a rule to make the icon inside the active button white: `.priority-btn.active i { color: #ffffff; }`."

--------------------------------------------------------------------------------
2. MONITORING PORTAL - COLUMN SORTING BUG
--------------------------------------------------------------------------------
* PHP Change:
  Fixed a critical sorting bug where an early `return createdA - createdB;` 
  bypassed other sorting keys, making all other sort selections (name, status, etc.) 
  unreachable dead-code.
* Agent Prompt for Python Version:
  "In `frontend/monitoring/script.js`, locate the `applySorting` function. 
  Find the early return statement `return createdA - createdB;` placed before the 
  sorting key evaluations. Move this `created_at` comparison logic to the very end 
  of the `.sort()` callback as a fallback tie-breaker when `valA === valB` so that 
  column-based sorting works correctly."

--------------------------------------------------------------------------------
3. DASHBOARDS - UNAUTHORIZED USER PAGE FLASH BUG
--------------------------------------------------------------------------------
* PHP Change:
  Added `style="display: none;"` inline on the root `<html>` elements in the 
  operator and monitoring HTML files. The JS unhides the page only after a 
  valid profile/session is fetched, preventing unauthorized layout flashing.
* Agent Prompt for Python Version:
  "In `frontend/operator/index.html` and `frontend/monitoring/index.html`, 
  add `style="display: none;"` to the `<html>` tag. This ensures that unauthorized 
  users navigating directly to the dashboards do not see a split-second flash of 
  the layouts before the async auth script redirects them back to the login page."

--------------------------------------------------------------------------------

5. BACKEND - KEEP CARDS FOR FORWARDED ITEMS
--------------------------------------------------------------------------------
* PHP Change:
  Modified the card validation and reset logic to include `forwarded` status. 
  This ensures that when a ticket is forwarded, its physical card number is 
  retained and not reallocated to new entries.
* Agent Prompt for Python Version:
  "In `backend/app/services/queue_services.py`, locate the functions `get_available_cards()` 
  and `reset_past_days_queue_items()`. Modify their filters to include `Status.FORWARDED` 
  alongside `Status.PENDING` and `Status.PROCESSING` so that forwarded tickets retain 
  their card numbers and do not return them to the available cards pool."

8. OPERATOR DASHBOARD - DYNAMIC FLASH / CALL NEXT ACTION
--------------------------------------------------------------------------------
* PHP Change:
  Consolidated the "Flash" and "Call Next" functionality. The Flash button changes
  its label dynamically to "Flash Next" if no queue item row is selected, and 
  "Flash" when a row is selected. Clicking it triggers a unified handler: if an
  item is selected, it flashes that item; otherwise, it triggers `/call-next` 
  to pull and call the next pending item in the queue.
* Agent Prompt for Python Version:
  "1. In `frontend/operator/index.html`, change the Flash button's ID to `flashBtn` 
     and set its click handler to `handleFlashAction()`.
   2. In `frontend/operator/script.js`, implement `handleFlashAction()`. If 
     `selectedEntryId` is set, call `flashSelected()`. Otherwise, call a new 
     asynchronous function `callNext()`.
   3. Implement `callNext()` to make a `PUT` request to `${API_BASE}/call-next`. 
     If successful, reload the queue. If it fails with a 404 or empty response, 
     alert that there are no pending items to call.
   4. Update `selectRow()` to change the text content of `flashBtn` to 'Flash', and 
     update `clearSelection()` and initial load to reset it to 'Flash Next'.
   5. In the backend (`backend/app/api/endpoints/queue.py` or equivalent), implement 
     a `PUT /call-next` route that retrieves the highest priority `PENDING` queue 
     item, transitions its status to `PROCESSING`, records its activation time, 
     saves it, and returns the item."

--------------------------------------------------------------------------------
* PHP Change:
  Added a "Hide Completed" toggle checkbox to filter out completed items from
  the queue table. Additionally, operator control panel buttons (Skip, Forward,

  on the status of the selected ticket, removing the need for manual alert checks.
* Agent Prompt for Python Version:
  "1. In `frontend/operator/index.html`, add a 'Hide Completed' checkbox with 
     id `hideCompleted` next to the filter dropdowns. Assign IDs to the control panel 
     buttons: `skipBtn`, `forwardBtn`, `completeBtn`, and `cancelBtn` (or `deferBtn`).
   2. In `frontend/operator/script.js`, implement `setPanelButtonState(buttonId, enabled)` 
     to set properties (`disabled`, `aria-disabled`), styling opacity, cursor, 
     and toggle the `is-disabled` class on the buttons.
   3. Implement `togglePanelButtons(status)` to enforce status-based permissions:
      - `processing`: enable skip, forward, complete, cancel; disable flash.
      - `pending`: enable flash, skip, forward, complete, cancel.
      - `forwarded`: enable flash, complete, cancel; disable skip, forward.
      - default (no selection, or completed/deferred status): disable all panel buttons.
   4. Call `syncPanelButtonsForSelection()` during initial load, row selection, and 
     clearing selection to ensure button states match the selected item's status.
   5. Update `applyQueueFilters()` to read the state of `#hideCompleted`. If checked, 
     exclude items whose status is 'completed' from the filtered queue output.
   6. Remove the redundant manual status checks and browser alerts from inside 
     `flashSelected()`, `skip()`, and `updateCurrent()` since invalid buttons 
     are now disabled."
